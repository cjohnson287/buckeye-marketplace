## 5. Code Quality Coaching (Non-Scoring)

- **Empty placeholder test**: [UnitTest1.cs](backend.Tests/UnitTest1.cs) contains an empty `[Fact]` with no assertions. While it passes, it adds no value. Consider removing it or replacing it with a real test to maintain a clean test suite.

- **Seeded credentials in source code**: [Program.cs](backend/Program.cs#L118-L119) hard-codes `admin@buckeye.local` / `Admin1234` and `student@buckeye.local` / `Student123` as seed data. While acceptable for an in-memory dev store, in a production scenario these should come from configuration or environment variables.

- **Missing input sanitization on admin product form**: [AdminDashboardPage.tsx](frontend/src/pages/AdminDashboardPage.tsx#L62-L90) — the product creation form sends user input directly. While the backend validates, adding frontend validation (e.g., non-empty title, positive price) would improve UX by catching errors before the network round-trip.

- **Category mapping hardcoded in admin page**: [AdminDashboardPage.tsx](frontend/src/pages/AdminDashboardPage.tsx#L100-L105) — `categoryMap` is a hardcoded `Record<string, string>` that must be manually kept in sync with the backend seed data. Consider fetching categories from a `GET /api/categories` endpoint.

- **Token stored in localStorage**: [http.ts](frontend/src/api/http.ts#L4-L6) — localStorage is vulnerable to XSS. For a more secure setup, consider using `httpOnly` cookies or at minimum pair localStorage with short token lifetimes and refresh token rotation.

## 6. Git Practices Coaching (Non-Scoring)

- **Commit granularity**: Consider keeping commits focused on single features or rubric items (e.g., "Add auth backend", "Add checkout flow") to make the git history easier to review and to isolate regressions.

- **Branch strategy**: Using feature branches (e.g., `feature/auth`, `feature/orders`) with pull requests — even on a solo project — builds habits that translate to professional team workflows.
